Hint-System für Buch-/Kurs-Puzzles

Konzept · vorberechnete, gestufte Tipps (Hint 1–3) je Puzzle · RookHub

Kernidee. Für jedes Buch-Puzzle werden einmalig im Voraus drei aufeinander aufbauende Tipps erzeugt und in der DB gespeichert. Beim Lösen tippt man auf „Tipp" und bekommt Stufe 1, nochmal → Stufe 2, nochmal → Stufe 3 — von vage zu konkret, ohne sofort die ganze Lösung zu verraten.

Quellen je Puzzle: die bereits verifizierte Lösungslinie (Moves ab StartPly), die vorhandenen Zug-Kommentare des Buches (Comment + MoveComments — bei Büchern wie „1001 Deadly Checkmates" oft sehr aussagekräftig) und optional eine Stockfish-Einschätzung. Daraus formuliert ein LLM (Claude) die drei Stufen — als Batch beim Import, nicht zur Laufzeit.

Warum vorberechnen statt on-demand?

✓ Vorberechnet (empfohlen)

  • Sofort & offline: Tipps liegen im BookPuzzleDto bei — kein Spinner, funktioniert mit dem Offline-Cache.
  • Kosten gedeckelt: 1× pro Puzzle, nie pro Nutzer/Versuch. Per Batch-API zusätzlich −50 %.
  • Kein API-Key im Request-Pfad: LLM läuft nur im Import, nicht bei jedem Klick.
  • Prüfbar: Tipps können vor dem Ausspielen reviewt/regeneriert werden.

✗ On-demand pro Klick

  • Latenz + Spinner bei jedem Tipp.
  • Kosten skalieren mit Nutzung, nicht mit Inhalt.
  • Geht offline gar nicht.
  • Backend müsste pro Request einen LLM-Call machen (Rate-Limits, Fehlerfälle).

→ Deine Präferenz (vorberechnen, Hint 1–3 je Puzzle) ist auch die technisch klar bessere. Der Rest des Dokuments baut darauf auf.

Die drei Stufen — die „Tipp-Leiter"

Jede Stufe deckt bewusst mehr auf. Stufe 1 + 2 verraten nie einen konkreten Zug; erst Stufe 3 nennt den ersten Zug bzw. die Schlüsselidee. Beispiel an einem Matt-Puzzle:

1

Motiv / Richtung — keine Felder, kein Zug

„Schwarz steht eng am Rand — achte auf die Grundreihe."

2

Welche Figur / welcher Bereich — noch kein konkreter Zug

„Deine Dame kann den entscheidenden Schlag führen; der schwarze König hat kein Luftloch."

3

Erster Zug bzw. Schlüsselidee — aber nicht die volle Variante

„Beginne mit Dame nach h8 — Matt folgt erzwungen."

Die „ganze Lösung" bleibt weiterhin der bestehende Mechanismus („Originallösung zeigen" / Durchspielen mit den Buch-Kommentaren). Die Tipp-Leiter ist der sanfte Weg dorthin.

Stockfish: brauchen wir's? optional

Heute gibt es nur Client-Stockfish (WASM), keinen Server-Engine. Für Puzzles ist die Moves-Linie aber bereits die geprüfte Lösung — das LLM braucht Stockfish nicht, um den richtigen Zug zu „finden". Stockfish liefert nur Zusatzsignal („Matt in 3", „erzwungen", Drohung), das die Tipp-Qualität schärft. Optionen:

  • Ohne Server-Engine (Start): LLM bekommt FEN + Lösungslinie + Kommentare. Reicht für gute Tipps, null neue Infrastruktur.
  • Mit einmaligem SF-Pass (später): kleines headless-Stockfish-Sidecar (Container) berechnet beim Import je Puzzle Eval/Mate-Distanz und gibt sie dem LLM mit. Nur Build-Zeit-Kosten, kein Laufzeit-Pfad.

Frontend (Solver)

  • Tipp-Button im book-puzzle-Solver (neben „wer am Zug" / Eval / Anzeigen). Progressive Enthüllung: 1. Tap → Stufe 1, 2. Tap → Stufe 2, 3. Tap → Stufe 3; danach Hinweis auf „Originallösung zeigen".
  • Nur wenn vorhanden: Button erscheint nur, wenn puzzle.hints?.length > 0 (Altbestand ohne Tipps zeigt ihn nicht).
  • Nutzung tracken wie vizShowCount: neues Feld an BookPuzzleAttempt (z.B. HintsUsed), fließt später in Statistik/Trainingsziele.
  • Modus-Konsistenz (CLAUDE.md-Regel): überlegen, ob Standard-/Endless-Puzzles denselben Button bekommen — dort gibt es aber keine Buch-Kommentare, also evtl. nur generisch oder bewusst nur Buch/Kurs.

Kosten — grob

Pro Puzzle ~300–800 Token Input (FEN + Linie + Kommentare) + ~150 Token Output. Bei 1.000 Puzzles, einmalig, mit Batch-API (−50 %):

Modell$/1M In/Out~1.000 Puzzles (Batch)Eignung
Haiku 4.5$1 / $5< ~$0,50Massengenerierung, einfache Matt-/Taktik-Tipps
Sonnet 4.6$3 / $15~$1–2guter Mittelweg
Opus 4.8$5 / $25~$2–4beste Tipp-Qualität bei kniffligen Stellungen

Selbst Opus über 1.000 Puzzles bleibt im einstelligen Dollar-Bereich — weil es eine Einmal-Aktion ist. Gewählt: Opus 4.8. Mit de/en/hr ~×3 (ein Call je Sprache) → grob ~$6–12 pro 1.000 Puzzles, weiterhin einmalig.

Qualität & Risiken

  • Kein Halluzinieren über die Stellung: dem LLM die verifizierte Lösungslinie geben und es nur beschreiben/abstufen lassen — nicht selbst „lösen". Das eliminiert die Hauptfehlerquelle.
  • Stufen-Leak verhindern: System-Prompt + JSON-Schema mit klarer Regel „Stufe 1+2 ohne konkreten Zug/Feld". Optional ein kurzer Verifikations-Pass (zweiter, billiger Call oder Regel-Check), der prüft, dass Stufe 1 keinen Zug nennt.
  • Comments-Qualität schwankt: manche Puzzles haben dichte Kommentare, andere gar keine — Tipps müssen auch ohne Kommentar (nur aus FEN+Linie) funktionieren.
  • Mehrsprachigkeit: RookHub ist de/en/hr. Entweder Tipps auf Deutsch generieren (Hauptpublikum) oder pro Sprache — Letzteres ×3 Kosten. Vorschlag: erst Deutsch, später optional übersetzen.
